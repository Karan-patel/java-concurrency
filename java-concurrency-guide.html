Section 01

Concurrency Design Patterns

8 patterns that form the vocabulary of concurrent system design. Mastering when and why to use each one separates architects from implementers. Every pattern here maps to a real production scenario at Google, Netflix, or any high-scale system.

1. Producer-Consumer

Decouple production rate from consumption rate via a shared buffer. The backbone of every async pipeline — prevents fast producers from overwhelming slow consumers and vice versa.

Java
💡 Senior Insight: Queue capacity = throughput × max acceptable latency (Little's Law). Unbounded queues cause silent OOM at 3am. Too small starves producers. Always size from measured production data, not intuition.

2. Thread Pool

Reuse threads, bound resource usage, enable lifecycle management. Thread creation costs ~1ms and ~1MB stack — never throw away threads you'll need again. The pool IS the backpressure mechanism.

Java

3. Active Object

Each object runs in its own thread with a private message queue. Calls become async messages — no shared state between objects, no locks needed within a single entity. Akka actors implement this pattern at scale.

Java
💡 When to use: Per-entity isolation — per-session, per-account, per-connection. Each entity owns its mailbox giving sequential execution per entity + maximum parallelism across entities. Zero contention within a single account.

4. Read-Write Lock

N concurrent readers OR 1 exclusive writer. Critical for read-heavy shared caches, configuration registries, and service directories. Most production data structures are 90%+ reads.

Java

5. Two-Phase Termination

Clean cooperative thread shutdown. The thread checks a termination flag and handles its own cleanup before dying. Never use Thread.stop() — it corrupts shared state silently.

Java

6. Thread-Per-Message

One thread per request/message. Trivially simple code — no callbacks, no state machines. Only viable at scale with Java 21 virtual threads (millions of cheap threads vs ~10k expensive platform threads).

Java

7. Scheduler Pattern

Control timing and ordering of task execution. The critical decision: scheduleAtFixedRate vs scheduleWithFixedDelay causes real production incidents when misunderstood.

Java
⚠️ Critical: scheduleAtFixedRate fires even if the previous run is still executing — tasks pile up silently. Use scheduleWithFixedDelay for tasks with variable or unpredictable duration (DB queries, HTTP calls, etc.).

8. Half-Sync / Half-Async

Async I/O layer + sync business logic layer. The design of Netty, Tomcat NIO, Node.js. The key insight: async I/O is efficient but hard to write; sync code is easy to write and test. Two layers collaborate via a handoff queue.

Java
Practice Design Patterns — Build It To Own It
  1. Producer-Consumer with 2 producers, 4 consumers, queue=100, graceful shutdown with poison pill per consumer. Verify no tasks are lost.
  2. Build an ActiveObject bank account: deposit/withdraw/getBalance — all serialized without any synchronized or Lock anywhere. Use CompletableFuture for getBalance return.
  3. Two-Phase Termination for a log-flushing thread that must complete its current flush before dying. Add a 5-second timeout with an alert if it doesn't stop.
  4. Hard: Rate-limited HTTP dispatcher: max 10 req/s, queue=1000, CallerRunsPolicy for backpressure. Verify with 50 concurrent producers at 30 req/s each.
Section 02

Java Memory Model

The contract between your code and the hardware. Every thread-safety bug traces to a JMM violation. Understand happens-before or you are guessing — and guessing in concurrent code means production incidents at 3am.

Why This Matters: CPU Caches & Reordering

Each CPU core has its own L1/L2 cache. Writes from Core 1 may not be visible to Core 2 for milliseconds. CPUs and JVMs reorder instructions for performance (out-of-order execution, write buffers, register caching). JMM defines exactly when one thread's writes are guaranteed visible to another.

Happens-Before: The One Rule That Governs Everything

Action A happens-before B means: A's effects are guaranteed visible when B executes. Key HB edges in Java:

  • Program order within a single thread (your statements run in order)
  • Monitor unlock HB → same monitor re-lock (this is how synchronized provides visibility)
  • Volatile write HB → subsequent volatile read of the same field
  • Thread.start() HB → any action inside the started thread
  • Thread.join() HB → return from join() in the joining thread
  • Static initializer HB → first access of the class by any thread
Java

volatile: Exactly What It Does and Doesn't

✓ volatile DOES
  • Guarantees visibility — flushes CPU caches on read/write
  • Prevents instruction reordering around the access
  • Makes long/double read/write atomic (otherwise two 32-bit ops)
  • Establishes happens-before on every access
✗ volatile DOES NOT
  • Make compound operations atomic (i++ is 3 separate ops)
  • Replace synchronized for check-then-act patterns
  • Provide mutual exclusion between threads
  • Handle atomicity across multiple variables
Java

An object is safely published if other threads that see its reference are also guaranteed to see its fully constructed state. Without this guarantee, another thread can see a partially initialized object — one of the most insidious bugs.

Section 03

synchronized & Intrinsic Locks

Every Java object is a monitor. synchronized is the simplest concurrency primitive — understand its internals, guarantees, and hard limits before reaching for anything else. Most concurrency bugs come from misusing the primitives you already know.

How the Monitor Works Under the Hood

Every Java object has a lock word in its header. Acquiring: CAS attempt first (fast path — succeeds when uncontended). On contention, falls back to OS futex (mutex) — this is the expensive slow path. JVM applies biased locking, thin locking, and fat locking optimizations. Critically: acquiring a lock also flushes CPU caches — this is the visibility guarantee you get for free.

Java

wait() / notify() / notifyAll()

Java
⚠️ Critical Rule: ALWAYS use while (!condition) wait() — never if. Spurious wakeups are real on Linux (POSIX allows them). notifyAll() is almost always correct; notify() is an optimization valid only when all waiters are identical and any one can make progress.

Reentrancy and Common Pitfalls

Practice synchronized & Monitors

  2. Classic deadlock: Thread A holds account1, wants account2. Thread B vice versa. Reproduce it (observe the deadlock), then fix with consistent lock ordering by account ID.
  3. Thread barrier for N threads using wait/notifyAll: all N must arrive before any proceeds. Test with 8 threads. What happens if one thread throws an exception?

Section 05

ReadWriteLock & StampedLock

Optimize for read-heavy workloads. ReadWriteLock: N concurrent readers OR 1 exclusive writer. StampedLock adds optimistic reads — 3–5× faster on read-dominated paths by skipping the lock entirely on the happy path.

ReentrantReadWriteLock

Java
⚠️ Writer Starvation: In read-heavy workloads, a continuous stream of readers prevents writers from ever acquiring. RRWL has some writer preference but it is incomplete under sustained reader pressure. If writer starvation is observable (measure it!), use StampedLock or a custom fair variant.

StampedLock: Optimistic Reads

Three modes: write lock (exclusive), read lock (shared), optimistic read (no lock acquired — validate after reading). Optimistic reads do not block writers at all — the fastest concurrency path possible in Java.

Java
ReentrantReadWriteLock
  • Reentrant — safe for recursive lock acquisition
  • Write → read downgrade supported
  • Condition variables available
  • Good for moderate concurrency, reentrant code
  • Reader count tracked — predictable
StampedLock
  • NOT reentrant — deadlock if a thread re-enters!
  • Optimistic reads — massive throughput gains
  • No condition variables
  • Best for read-dominant, no recursive locking
  • Stamp invalidation gives clean consistency model

  3. Medium: Simulate writer starvation: 10 reader threads running at 100% CPU, 1 writer thread. Measure average writer wait time. What's the fix?
  4. Hard: Thread-safe sorted list with RRWL. Reads dominate (contains, size). Benchmark RRWL vs StampedLock. Under what contention level does StampedLock win? When does RRWL's overhead become visible?

Atomic Operations & CAS

Lock-free programming via hardware Compare-And-Swap. Faster than locks for single-variable state because they avoid OS scheduling and kernel calls. The foundation of all non-blocking data structures in java.util.concurrent.

Compare-And-Swap: The Hardware Primitive

CAS = "If memory[addr] == expected, write desired atomically, return true; otherwise return false." Single CPU instruction (LOCK CMPXCHG on x86). No OS mutex — pure hardware atomicity with memory barrier semantics.

Java — CAS loop pattern

ABA Problem and LongAdder

Java
💡 AtomicLong vs LongAdder: Use AtomicLong when you need to read the current value and CAS on it (IDs, state machines). Use LongAdder for counters and metrics under high concurrency — it distributes updates to per-thread cells and merges on sum(), dramatically reducing cache line contention. Difference: 10x–50x fewer cache misses under 8+ threads.

Thread Pools & ExecutorService

Thread creation costs ~1ms and ~1MB stack. Pools amortize this cost. ThreadPoolExecutor is the real engine behind everything — every Executors.newXxx() is just a convenience shortcut with hidden dangerous defaults that have caused countless production OOM incidents.

ThreadPoolExecutor: The Full Control Surface

Java — Production-grade ThreadPoolExecutor
💡 Submission lifecycle: (1) Below core? → new core thread. (2) At core capacity? → enqueue. (3) Queue full and below max? → new non-core thread. (4) At max AND queue full? → rejection handler fires. Max threads are created ONLY when the queue is full — a critical subtlety that surprises engineers.

Pre-built Pools: What They Actually Are

Java
⚠️ Never in Production: newFixedThreadPool(n) uses an unbounded LinkedBlockingQueue — under sustained load the queue grows forever, consuming gigabytes of heap. No warnings. No errors. Just OOM at 3am. Always construct ThreadPoolExecutor directly with a bounded queue in production.

Synchronizers

Pre-built coordination structures for specific patterns. Each solves exactly one problem with precisely the right semantics. Always prefer these over hand-rolled coordination with wait/notify — they are more readable, more correct, and battle-tested.

CountDownLatch — One-Shot Gate

N events must happen (in any order, by any thread), then blocked thread(s) proceed. Cannot be reset. Perfect for "wait for N services to start" or "wait for N workers to finish".

Java

CyclicBarrier — Reusable Rendezvous

All N participating threads must arrive at the barrier point, then all proceed together. Reusable across phases. Ideal for phased parallel computation — map phase, then reduce phase, then combine.

Java
CountDownLatch
  • One-shot — not reusable after reaching zero
  • Any thread can call countDown()
  • Waiters are different threads from decrementers
  • Semantics: "wait for N events to happen"
  • Use for: startup, shutdown, one-time coordination
CyclicBarrier
  • Reusable — resets after each barrier trip
  • All N participants must call await()
  • Same threads are both workers and waiters
  • Semantics: "all N meet here, then go together"
  • Use for: phased parallel algorithms

Fork/Join Framework

Divide-and-conquer parallelism with work-stealing. The engine behind parallel streams. Use when tasks recursively decompose and subtask sizes are uneven or unpredictable — work-stealing automatically balances the load across all cores.

Work-Stealing: The Core Insight

Each thread has its own double-ended deque. Own tasks pushed and popped from the top (LIFO — cache-friendly, hot data). Idle threads steal from the bottom of other threads' deques (FIFO — steals the oldest, largest chunk of work). No central queue, no central lock. This is why it automatically handles wildly uneven task sizes.

Java — RecursiveTask (returns value)
💡 Threshold Tuning: Too small → task creation and scheduling overhead dominates. Too large → poor load balancing, some threads idle while others grind. Target base case execution time of 0.5–1ms. For array work, divide total size by (N_cpus × 4) as a starting point, then profile.

RecursiveAction and Parallel Streams

Java
⚠️ Never Block in commonPool: Blocking inside ForkJoinPool.commonPool() starves ALL parallel streams application-wide. On an 8-core machine, one blocked thread = 12.5% throughput loss. For any blocking I/O inside parallel streams, always use a custom ForkJoinPool with extra threads equal to expected blocking factor.
