Array

🧬 Core DNA

Array is a contiguous block of memory divided into equal-sized slots, where each slot is identified by an index. Think of it as a row of numbered lockers, each exactly the same size, lined up next to each other with no gaps.

Mental Model: A parking lot with numbered spaces. Each space is the same size, you know exactly where space #5 is (5 spots from the start), and you can drive directly to it. But you can't easily add more spaces in the middle without moving all cars after it.
[0] [1] [2] [3] [4] [5]
A B C D E F
Memory: →→→→→→ (continuous)

O(n)
Delete

Space
O(n)

* O(1) amortized if dynamic array with capacity management

✓ Strengths
  • Lightning-fast random access - Get any element in constant time using index
  • Cache-friendly - Contiguous memory = CPU cache loves it
  • Memory efficient - No extra pointers, just raw data
  • Simple and universal - Every language has arrays built-in
  • Predictable memory layout - Easy to reason about and debug
✗ Limitations
  • Fixed size - Once created, size is fixed (unless dynamic array with reallocation)
  • Expensive insertions/deletions - Must shift all elements after the position
  • Contiguous memory required - Large arrays need one big continuous block
  • Wasted space - Dynamic arrays often over-allocate to avoid frequent resizing
  • Not suitable for frequent modifications - Middle insertions/deletions kill performance
🎯 When to Use
Known, fixed-size collections: Days of week, RGB color values, matrix operations
Frequent random access: Looking up elements by index constantly (game board, pixel data)
Sequential processing: Iterating through all elements in order
Performance-critical code: Sorting algorithms, numerical computations where cache locality matters
Buffer implementations: Circular buffers, string buffers, I/O buffers

💡 Key Insight

Arrays are the foundation of computing. Their power comes from mathematics: address = base + (index × element_size). This simple formula enables O(1) access. The tradeoff is rigidity - that contiguous memory that makes them fast also makes them inflexible. Use arrays when you know the size and need speed; avoid when you need frequent insertions/deletions in the middle.

Linked List

🧬 Core DNA

A linked list is a chain of nodes scattered in memory, where each node contains data and a pointer to the next node. Like a treasure hunt where each clue tells you where the next clue is hidden. There's no indexing magic - you must follow the chain.

Mental Model: A train where each car (node) carries cargo (data) and a coupling (pointer) that connects to the next car. To get to car #10, you must walk through cars 1-9. But you can easily insert a new car anywhere or detach cars by just changing couplings.
[A|→] → [B|→] → [C|→] → [D|→] → NULL
Memory: scattered, not continuous
Each node: {data, next pointer}

⏱️ Time & Space Complexity

Access
O(n)
Search
O(n)
Insert (head)
O(1)
Insert (middle)
O(n)
Delete (known node)
O(1)
Space
O(n)

Note: Each node needs extra space for pointer(s)

✓ Strengths
  • Dynamic size - Grows and shrinks easily, no reallocation needed
  • Efficient insertions/deletions - O(1) at head, or when you have the node reference
  • No memory waste - Uses exactly what you need (plus pointer overhead)
  • No contiguous memory needed - Can use fragmented memory spaces
  • Easy restructuring - Can rearrange nodes by just changing pointers
✗ Limitations
  • Slow random access - Must traverse from head to reach any element
  • Extra memory for pointers - Each node needs additional space for next (and prev in doubly-linked)
  • Cache-unfriendly - Nodes scattered in memory = poor cache locality
  • No backward traversal - In singly-linked list, can't go backwards
  • More complex - Pointer management, edge cases (null checks, head/tail)
🎯 When to Use
Frequent insertions/deletions: Task lists where items are constantly added/removed
Unknown size: When you don't know how many elements you'll have
Implementation of other structures: Base for stacks, queues, hash table chaining
Undo functionality: Browser history, text editor undo/redo
Music/video playlists: Where next/previous navigation and insertion are common

💡 Key Insight

Linked lists trade access speed for modification flexibility. While arrays excel at "I need element #5", linked lists excel at "insert this new element here" or "remove this element". The fundamental difference: arrays use arithmetic (calculate address), linked lists use references (follow pointers). Choose based on your primary operation pattern.

Stack

🧬 Core DNA

A stack is a LIFO (Last In, First Out) structure - like a stack of plates. You can only add or remove from the top. The most recent item added is the first one to be removed. It's a restricted access data structure with only two main operations: push (add to top) and pop (remove from top).

Mental Model: A stack of books or a cafeteria tray dispenser. You add new books on top, and when you need one, you take from the top. You can't grab the bottom book without removing all the ones above it. Last one you placed is the first one you'll pick up.

✓ Strengths
  • Simple and intuitive - Easy to understand and implement
  • Constant-time operations - All primary operations are O(1)
  • Memory efficient - Can be implemented with array or linked list efficiently
  • Natural for recursion - Models the call stack perfectly
  • Useful for backtracking - Naturally supports undo operations
✗ Limitations
  • No random access - Can't access middle elements without popping
  • Limited operations - Only push, pop, peek - very restricted
  • Fixed order - LIFO only, can't process oldest first
  • Stack overflow risk - Limited size in fixed implementations
  • Not suitable for most collections - Too restrictive for general-purpose use
🎯 When to Use
Function call management: Every program uses call stack for function invocation
Expression evaluation: Converting infix to postfix, evaluating mathematical expressions
Undo mechanisms: Text editors, photo editing software - track operations in reverse order
Browser history: Back button navigation (though modern browsers use more complex structures)
Syntax parsing: Checking balanced parentheses, XML/HTML tag matching
Backtracking algorithms: DFS traversal, maze solving, puzzle solving

💡 Key Insight

Stacks are the essence of "most recent first". They're not a general-purpose structure - they're a specialized tool for specific patterns. The restriction (LIFO only) is actually their power: it enforces a discipline that naturally fits problems involving reversal, backtracking, or nested structures. When you see recursion, think stack. When you need undo, think stack.

LinearFIFO

Queue

🧬 Core DNA

A queue is a FIFO (First In, First Out) structure - like a line at a ticket counter. You add at the back (enqueue) and remove from the front (dequeue). The oldest item is served first. It's a fair, ordered structure where everyone waits their turn.

Mental Model: A checkout line at a grocery store. First person to arrive is first to be served. New people join at the back. No cutting in line! The person at the front has been waiting longest and gets served first. Fair and orderly.
FRONT ← [A][B][C][D] ← REAR

↓ ↓
FIFO: A → B → C → D

Search
O(n)
Space

✓ Strengths
  • Fair ordering - First come, first served - no priority jumping
  • Constant-time operations - Enqueue and dequeue both O(1)
  • Predictable behavior - Order is maintained, no surprises
  • Natural for scheduling - Models real-world waiting scenarios
  • Simple to implement - Can use array (circular) or linked list
✗ Limitations
  • No random access - Can't access middle elements
  • Limited operations - Only enqueue, dequeue, peek
  • No priority - Can't handle urgent items (need priority queue)
  • Implementation choice matters - Array-based can waste space, linked list uses extra memory
  • Order strictly enforced - Can't reorder or reorganize once queued
🎯 When to Use
Task scheduling: CPU process scheduling, print job queue, task queues in systems
BFS traversal: Breadth-first search in graphs and trees - level-by-level exploration
Message systems: Message queues, event handling, asynchronous processing
Buffer management: I/O buffers, streaming data, keyboard buffer
Request handling: Web server request queue, API rate limiting
Simulation: Modeling real-world queues (bank, restaurant, call center)

💡 Key Insight

Queues embody fairness through ordering. Unlike stacks (LIFO = most recent first), queues are FIFO = oldest first. This simple principle makes them perfect for any scenario where order matters and fairness is desired. The key distinction from stack: stack is about reversal (recursive unwinding), queue is about preservation of order (process in arrival sequence).

Hash Table (Hash Map)

🧬 Core DNA

A hash table uses a hash function to map keys to array indices, enabling near-instant lookups. It's magic: give it a key (any data type), the hash function converts it to an array index, and boom - direct access. Like a library catalog that instantly tells you the exact shelf where your book sits.

Mental Model: A library with a magical librarian. You give the librarian a book title, they instantly compute which shelf it's on (hash function), and point you directly there. No searching through every shelf. The magic is in the hash function that converts any title into a shelf number.
Key → HASH FUNCTION → Index

"John" → hash() → 3
"Mary" → hash() → 7
"Bob" → hash() → 3 (collision!)

[0] → null
[1] → null
[2] → null
[3] → "John"→"Bob" (chain)
[4] → null
...

⏱️ Time & Space Complexity

Insert
O(1) avg
Search
O(1) avg
Delete
O(1) avg
Worst Case
O(n)
Space
O(n)

Worst case O(n) occurs with many collisions or poor hash function

✓ Strengths
  • Lightning-fast lookups - Average O(1) for insert, search, delete
  • Key-value pairs - Natural for dictionaries, maps, caches
  • Flexible keys - Can use strings, objects, or any hashable type as keys
  • Excellent for counting/frequency - Track occurrences, build histograms
  • Deduplication - Automatically handles unique keys
✗ Limitations
  • No ordering - Elements are not stored in any particular order
  • Hash collisions - Poor hash function or high load factor degrades performance
  • Space overhead - Needs extra space for load factor and collision handling
  • No range queries - Can't efficiently get "all keys between X and Y"
  • Hash function quality matters - Bad hash = bad performance
  • Not cache-friendly - Random access pattern, unlike arrays
🎯 When to Use
Caching: Memoization, DNS lookup, web browser cache - fast key-based retrieval
Counting/frequency: Word count, character frequency, duplicate detection
Database indexing: Quick lookups by primary key
Symbol tables: Compilers, interpreters - variable name to value mapping
Set operations: When you need unique collection with fast membership testing
Two-sum problems: Classic interview problems where you need complement lookup

💡 Key Insight

Hash tables are the speed hack of data structures. They trade memory and ordering for incredible speed. The magic is mathematical: a good hash function distributes keys uniformly, turning search from O(n) to O(1). But remember: average case ≠ worst case. With bad luck (or bad hash function), you can degrade to O(n). Still, for key-value lookups, nothing beats a well-implemented hash table.

Tree (Binary Search Tree)

🧬 Core DNA

A tree is a hierarchical structure with nodes connected by edges, where each node has at most one parent and zero or more children. BST adds ordering: left subtree < node < right subtree. It combines the search efficiency of binary search with dynamic insertion/deletion.

Mental Model: A family tree or corporate org chart. One root (CEO), branches to children (managers), who have their own children (employees). In BST, add a rule: left child is always smaller, right child always larger. To find someone, you make decisions at each level, eliminating half the remaining possibilities.

3 10

Left < Parent < Right

Search (balanced)
O(log n)
Insert (balanced)
O(log n)
Delete (balanced)
O(log n)
Search (worst)
O(n)
Space
O(n)

Worst case when tree becomes skewed (like a linked list)

✓ Strengths
  • Logarithmic operations - O(log n) search/insert/delete when balanced
  • Maintains order - In-order traversal gives sorted sequence
  • Dynamic size - Grows and shrinks without reallocation
  • Hierarchical relationships - Natural for representing nested structures
  • Range queries - Can efficiently find all elements in a range
  • No duplicates handling - Easy to enforce uniqueness or handle duplicates
✗ Limitations
  • Can become unbalanced - Degenerates to O(n) if insertions are sorted
  • Complex deletion - Removing nodes (especially with 2 children) is tricky
  • No O(1) operations - Everything requires traversal
  • Extra space for pointers - Each node needs left, right, (optionally parent) pointers
  • Not cache-friendly - Nodes scattered in memory
  • Balancing complexity - Self-balancing trees (AVL, Red-Black) add implementation complexity
🎯 When to Use
Sorted data with insertions/deletions: When you need to maintain sorted order and frequently modify
Range queries: Database indices, finding all elements between X and Y
Priority scheduling: When priorities change and you need to maintain order
File systems: Directory structures, hierarchical data
Expression parsing: Abstract syntax trees in compilers
Autocomplete/spell check: Tries (special trees) for prefix matching

💡 Key Insight

Trees are about divide and conquer through hierarchy. BST's power comes from binary search property: at each node, you eliminate half the tree. But here's the catch: balance is everything. A balanced BST is O(log n), a skewed BST is O(n) - same structure, vastly different performance. Real-world use usually demands self-balancing trees (AVL, Red-Black) to guarantee performance.

Tree Variants Comparison

Type Balance Guarantee Use Case

B-Tree Balanced multi-way tree Databases, file systems

Heap (Binary Heap)

🧬 Core DNA

A heap is a complete binary tree where each parent is greater/smaller than its children. Max-heap: parent > children. Min-heap: parent < children. Unlike BST, there's NO ordering between left and right children. The only guarantee: the root is always the max/min element.

Mental Model: A company hierarchy where everyone knows their boss is more important than them, but siblings (same level) have no ranking among themselves. The CEO (root) is the most important person. To find the most important person: just look at the root. No complex searching needed.
Max-Heap:
100
/ \
19 36
/ \ / \
17 3 25 1

Array: [100,19,36,17,3,25,1]
Parent > Children (no L/R order)

⏱️ Time & Space Complexity

Find Max/Min
O(1)
Insert
O(log n)
Extract Max/Min
O(log n)
Heapify
O(n)
Space
O(n)
✓ Strengths
  • Instant max/min access - O(1) to peek at highest priority element
  • Efficient priority operations - O(log n) insert and extract
  • Array implementation - Can be stored in array without pointers (cache-friendly!)
  • In-place heapify - Can turn unsorted array into heap in O(n)
  • Space efficient - No wasted space, no pointer overhead
✗ Limitations
  • No general search - Finding arbitrary element is O(n)
  • Only root is guaranteed - Can't efficiently access 2nd largest, 3rd largest, etc.
  • Not fully sorted - Elements are partially ordered, not fully sorted
  • Can't iterate in sorted order - Must extract elements to get sorted sequence
  • No range queries - Can't efficiently find "all elements between X and Y"
🎯 When to Use
Priority queues: Task scheduling, CPU scheduling, event simulation
Heap sort: Sorting algorithm with O(n log n) and O(1) space
Finding K largest/smallest: Top K problems, median finding
Graph algorithms: Dijkstra's shortest path, Prim's MST
Streaming data: Maintaining top N items from a stream
Merge K sorted lists: Efficiently merge multiple sorted sequences

💡 Key Insight

Heaps solve one specific problem brilliantly: "Give me the highest/lowest priority item quickly, repeatedly." They're not for sorting (though heap sort exists), not for searching, not for maintaining full order. They're for priority access. The genius is storing a tree in an array: parent = i, left_child = 2i+1, right_child = 2i+2. This makes heaps both fast AND cache-friendly - a rare combination.

Graph

🧬 Core DNA

A graph is a collection of vertices (nodes) connected by edges. Unlike trees, graphs can have cycles, multiple paths between nodes, disconnected components, and edges can have direction (directed) or not (undirected). It's the most general relationship structure - everything else is a special case of a graph.

Mental Model: A social network or road map. People (vertices) are connected by friendships (edges). You can follow multiple paths to reach someone. There might be cycles (mutual friends who know each other). Some people might be unreachable (disconnected components). Relationships can be one-way (directed) or two-way (undirected).
Undirected Graph:

V = vertices, E = edges. Complexities vary by representation.

✓ Strengths
  • Models complex relationships - Can represent any relationship structure
  • Rich algorithms available - Shortest path, connectivity, flow, coloring, etc.
  • Flexible - Can be directed/undirected, weighted/unweighted, cyclic/acyclic
  • Natural for networks - Social networks, computer networks, transportation
  • Multiple representations - Choose adjacency list or matrix based on needs
✗ Limitations
  • Complex to implement - More moving parts than other structures
  • Space-hungry - Adjacency matrix is O(V²) even for sparse graphs
  • Expensive operations - Many graph algorithms are O(V+E) or worse
  • No implicit ordering - Must use algorithms to find paths, order, etc.
  • Cycles complicate traversal - Need visited tracking to avoid infinite loops
  • Representation choice matters - List vs matrix affects performance significantly
🎯 When to Use
Social networks: Friend connections, followers, recommendations
Maps and navigation: Cities as vertices, roads as edges, GPS routing
Web crawling: Pages as vertices, links as edges
Dependency resolution: Package managers, build systems, task scheduling
Network routing: Computer networks, internet routing protocols
Recommendation systems: User-item relationships, collaborative filtering

💡 Key Insight

Graphs are the ultimate generalization. Trees are graphs (connected, acyclic). Linked lists are graphs (linear path). But graphs can model arbitrary relationships. The tradeoff: power comes with complexity. Simple graph operations are expensive (traversal = O(V+E)). You don't use graphs for simple ordered lists - you use them when relationships are complex and irregular. Choose adjacency list for sparse graphs, matrix for dense graphs.

Trie (Prefix Tree)

🧬 Core DNA

A trie is a tree where each node represents a character, and paths from root to nodes form strings. It's optimized for prefix-based operations. Common prefixes are shared, so "cat", "car", "card" share the path "c-a". Perfect for autocomplete, spell check, and word games.

Mental Model: A dictionary organized by letter structure. Start at the root, follow letters one by one. All words starting with "CAT" share the C-A-T path. To find all words starting with "CA", just traverse to "CA" and explore all branches below. Shared prefixes = shared paths = space savings.
root
/ | \
c d t
/ \ \
a o o
/ \ \
t r g
(cat)(car) (dog,to)

Stores: cat, car, dog, to

Insert
O(m)

O(m)

O(n×m)

m = length of string, n = number of strings

✓ Strengths
  • Blazing prefix search - Find all words with prefix "pre" instantly
  • Shared prefixes - Common prefixes stored once, saving space
  • Predictable performance - O(m) for string length m, independent of # words
  • No hash collisions - Deterministic, no randomness
  • Ordered traversal - Can get alphabetically sorted words
  • Wildcards and patterns - Can support regex-like matching
✗ Limitations
  • Space intensive - Each node needs array/map of children (typically 26 for lowercase)
  • Pointer overhead - Many pointers, especially in sparse tries
  • Cache-unfriendly - Pointer-chasing through scattered nodes
  • Only for strings - Not general-purpose, string-specific structure
  • Implementation complexity - More complex than hash table for simple string storage
  • Not always faster - Hash table might be faster for exact lookups
🎯 When to Use
Autocomplete: Search bars, text prediction, IDE code completion
Spell checker: Finding similar words, suggesting corrections
IP routing: Longest prefix matching in routers
Dictionary/word games: Scrabble, crossword solvers, anagram finders
T9 predictive text: Phone keyboards, text suggestions
String matching: Pattern matching, substring search in sets of strings

💡 Key Insight

Tries are specialized tools for prefix problems. If your problem involves "find all words starting with...", trie is your friend. The key insight: by organizing strings by character structure, you get prefix search for free. The cost: lots of nodes and pointers. Hash tables beat tries for exact match, but tries win decisively for prefix operations. It's a classic space-time tradeoff: spend space on structure to gain time on prefix queries.

💡 Master Principle

No single data structure is "best" - each optimizes different operations at the expense of others. Arrays trade flexibility for speed. Linked lists trade speed for flexibility. Hash tables trade order for lookup speed. Trees balance multiple operations. Understanding these tradeoffs IS data structures mastery. The expert doesn't pick the "best" structure; they pick the structure whose strengths match the problem's requirements.

Master data structures by understanding their DNA - the fundamental principle that drives all their properties.

Remember: Every advantage has a corresponding disadvantage. Choose wisely based on your problem's needs.
